# Exeligmos Sync Server

Run this on a Mac/PC on the same Wi-Fi network as the iPhone.

```sh
cd sync-server
npm start
```

Open the printed URL in a browser to browse the synced folder state. In the iOS app, go to Settings -> LAN Sync, enter the same URL, then use:

- Test server connection
- Sync phone state
- Sync phone state now
- Auto sync phone state
- Restore from server folders

Records are stored under `sync-server/data/` as regular folders. Override with:

```sh
EXELIGMOS_SYNC_DATA=/path/to/exeligmos-data npm start
```

The app state is intentionally stored as small independent files:

```text
data/
  tags/<tag-id>/tag.json
  entries/<unix>_<closest-saros>_<octal-phase>/entry.json
  entries/<unix>_<closest-saros>_<octal-phase>/media.json
  entries/<unix>_<closest-saros>_<octal-phase>/media/<files>
```

The phone is the source of truth. Each sync first mirrors the phone's current tag list and entry IDs into `tags/` and `entries/`: tags are rewritten because they are small and editable, and server tags/entries that are no longer present on the phone are deleted. Missing entries are then uploaded one at a time so the app never has to build one huge media payload in memory.

The server does not write a large `payload.json`. Browser views, manifests, and restore payloads are generated by scanning this folder hierarchy. Each entry lives in one folder. The folder name contains the Unix timestamp, the Saros number of the closest flip, and that Saros octal phase; the entry JSON still carries all associated Saros numbers for filtering.

Troubleshooting:

- Open `http://<computer-ip>:8787/health` from the phone browser. It should show JSON with `ok: true`.
- Open `http://<computer-ip>:8787/api/manifest` to see which tag, entry, and media IDs are already on the server.
- If the app says `HTTP 404` with `No synced records have been uploaded yet`, upload from the app before restoring.
- If the browser downloads an empty file, make sure this Node server is running with `npm start`, not a generic static file server on the same port.
- Sync uploads media only for entries that changed or have missing media on the server. If one very large video still hits `HTTP 413`, start the server with a larger limit, for example `MAX_BODY_BYTES=4294967296 npm start`.
